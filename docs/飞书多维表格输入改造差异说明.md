# 飞书多维表格输入改造差异说明

更新时间：2026-06-30

## 1. 能力定位

当前项目已经具备“飞书 Bot 收图 -> 本地压缩 -> AI 识图 -> 写入素材清单 -> DeerFlow / Dashboard 控制”的基础能力。

你现在提出的新方向不是继续让用户把图片直接发给飞书 Bot，而是：

> 以飞书多维表格作为图片资产的统一输入源，由本项目从多维表格拉取图片附件和字段信息，下载到本地后完成压缩、识别、沉淀，并生成适合 PPT 调用的素材描述。

因此，后续目标应从“飞书聊天收图工具”升级为“飞书多维表格驱动的 PPT 图片资产沉淀系统”。

## 2. 你刚刚提出的新输入设计

### 2.1 多维表格字段

飞书多维表格至少会提供这些字段：

| 字段 | 示例 | 含义 |
| --- | --- | --- |
| 名字 | minduck图标 | 素材名称，优先作为素材主标题 |
| 描述 | 2026-minduck图标；董办-xxx生活照片 | 人工输入的背景说明、时间、特殊备注 |
| 文件 | 图片.png | 图片附件，项目需要下载到本地 |
| 来源 | 下载 / 拍摄 / 截图 / AI | 图片来源类型 |
| 是否可用 | y / n | 表格中的业务字段，不是项目审核流程 |

重要修正：

- `是否可用` 不是 Dashboard 人工审核入口。
- `是否可用` 是多维表格已经提供的字段。
- 项目只读取该字段并按规则处理，不由 AI 决定是否可用。

### 2.2 图片分类

后续图片分类建议固定为四类：

| 分类 | 说明 | 例子 |
| --- | --- | --- |
| 背景 | 底图、氛围图、纹理图、空间图 | PPT 封面、章节页背景 |
| 内容 | 产品、肖像、截图、实物照片 | 产品界面、人物生活照、软件截图 |
| 概念 | logo、符号、比喻图、抽象图 | Minduck 图标、品牌符号、概念隐喻 |
| 信息 | 流程、架构、图表、信息图 | 架构图、流程图、数据图表 |

### 2.3 图片生命周期

目标流程：

```text
上传到飞书多维表格
  -> 项目拉取多维表格记录
  -> 下载图片附件到本地
  -> 压缩
  -> AI 结合图片 + 表格字段识别
  -> 沉淀为本地资产
  -> 写回多维表格处理结果
  -> 后续 PPT 调用本地资产
```

人的参与：

```text
上传 -> 使用
```

AI / 项目的参与：

```text
拉取 -> 下载 -> 压缩 -> 识别 -> 分类 -> 沉淀 -> 写回
```

## 3. 当前实现是什么

### 3.1 当前输入源

当前项目主要输入源是飞书 Bot 聊天图片：

```text
同事把图片发给飞书 Bot
  -> 图片进入 image_compressor/images_raw/
```

相关说明见当前 README：

```text
飞书 Bot 自动接收图片
图片直接进入 images_raw
```

当前飞书相关代码主要是：

| 文件 | 当前职责 |
| --- | --- |
| `python/feishu_client.py` | 发送飞书通知、获取 tenant token、发送文本消息 |
| `image_compressor/images_raw/_feishu_sources.jsonl` | 记录飞书聊天收图来源 |
| `deer-flow/backend/app/channels/feishu.py` | DeerFlow 飞书渠道能力 |

当前没有实现：

- 从飞书多维表格读取记录
- 从多维表格附件字段下载图片
- 把处理结果写回多维表格记录
- 监控多维表格容量

### 3.2 当前处理流程

当前 runner 固定流程在 `python/deerflow_runner.py` 和 `python/yqs_deerflow_tools.py` 中：

```text
检查 images_raw
  -> 压缩到 images_compressed
  -> AI 识图 images_compressed
  -> 写入 案例素材清单_表格.md
  -> 写入 案例素材清单.xlsx
  -> 写入 素材分辨率.csv
  -> 移动成功识图图片到 images_recognized
```

当前目录：

| 路径 | 当前用途 |
| --- | --- |
| `image_compressor/images_raw/` | 飞书 Bot 收到的原图，等待压缩 |
| `image_compressor/images_compressed/` | 压缩后的待识图图片 |
| `image_compressor/images_recognized/` | AI 识图成功后的图片 |
| `case_materials/` | 现有正式素材库 |
| `runtime/processed_files.json` | 已处理文件记录 |
| `runtime/recognition_state.json` | AI 识图状态 |
| `runtime/deerflow_agent_state.json` | Dashboard / DeerFlow 托管状态 |

### 3.3 当前 AI 识别输出

当前规则文件是：

```text
rules/image_recognition.md
```

当前 prompt 仍包含“是否适合进入正式素材库”“如果不适合，说明原因和处理建议”等审核式表达。

后续要改成 PPT 调用导向：

```text
适合用于 xxx
可用于 xxx PPT 页面
推荐用于 xxx 场景
```

不再把“图片审核”作为核心功能。

## 4. 你的新目标与当前实现的区别

| 维度 | 当前实现 | 新目标 |
| --- | --- | --- |
| 输入源 | 飞书 Bot 聊天收图 | 飞书多维表格记录 |
| 输入字段 | 基本只有图片文件和来源记录 | 名字、描述、文件、来源、是否可用 |
| 图片进入本地方式 | Bot 收图直接保存到 `images_raw` | 项目主动从多维表格下载附件 |
| 是否可用 | 当前由 AI prompt 评估“是否入库”倾向 | 多维表格字段提供，项目读取并尊重 |
| AI 描述口吻 | 存在“不建议使用 / 不适合”审核式描述 | 默认生成“适合用于 xxx”的 PPT 调用描述 |
| 处理结果写回 | 写本地 MD / XLSX / CSV | 写本地文件，同时写回飞书多维表格 |
| 告警 | 目前主要是状态文件和飞书完成通知 | 需要容量、超时、拉取量过大等告警 |
| 数据规模 | 本地队列为主 | 要考虑飞书多维表格 20000 上限 |
| 后续扩展 | Dashboard / DeerFlow 控制流程 | 数据量大后支持子表拆分 |

## 5. 需要新添加的内容

### 5.1 飞书多维表格客户端

新增一个独立模块，例如：

```text
python/feishu_bitable_client.py
```

职责：

- 获取飞书 tenant access token
- 读取多维表格记录
- 按字段名解析记录
- 下载附件字段里的图片
- 更新记录字段
- 查询表格记录总量，用于容量告警

建议不要把这部分直接塞进现有 `python/feishu_client.py`。

原因：

- 现有 `feishu_client.py` 是飞书消息通知客户端。
- 多维表格 API 是另一类能力，字段、附件、记录更新都不同。

### 5.2 多维表格同步模块

新增：

```text
python/bitable_sync.py
```

职责：

- 拉取“待处理”的多维表格记录
- 校验字段完整性
- 下载图片附件到本地
- 保存表格 metadata 映射
- 根据 `是否可用` 决定是否进入后续处理
- 将处理状态写回多维表格

建议本地 metadata 文件：

```text
runtime/bitable_records.json
runtime/bitable_file_map.json
```

示例结构：

```json
{
  "20260630_xxx.png": {
    "record_id": "recxxxx",
    "名字": "minduck图标",
    "描述": "2026-minduck图标",
    "来源": "下载",
    "是否可用": "y",
    "原始附件名": "图片.png",
    "本地原图路径": "image_compressor/images_raw/20260630_xxx.png"
  }
}
```

### 5.3 多维表格处理字段

建议在飞书多维表格新增系统字段：

| 字段 | 用途 |
| --- | --- |
| 处理状态 | 待处理 / 处理中 / 已完成 / 已跳过 / 失败 |
| 本地原图路径 | 下载到本地后的 raw 路径 |
| 本地压缩图路径 | 压缩后的路径 |
| 本地沉淀路径 | 最终可调用资产路径 |
| AI识别描述 | AI 对画面的结构化描述 |
| PPT适用场景 | 适合用于哪些 PPT 页面 |
| 图片分类 | 背景 / 内容 / 概念 / 信息 |
| 关键词 | 检索用标签 |
| 错误信息 | 下载、压缩、识图、写回失败原因 |
| 处理时间 | 最近处理时间 |

### 5.4 本地资产沉淀目录

当前 `case_materials/` 已存在，但识图成功图片目前主要移动到：

```text
image_compressor/images_recognized/
```

后续建议新增稳定资产目录，例如：

```text
case_materials/assets/
case_materials/assets/背景/
case_materials/assets/内容/
case_materials/assets/概念/
case_materials/assets/信息/
```

`images_recognized` 可以继续作为处理完成区，`case_materials/assets` 作为 PPT 调用资产库。

### 5.5 告警模块

新增：

```text
python/alerts.py
```

职责：

- 多维表格容量告警
- AI 识别超时告警
- 单次拉取量过大告警
- 附件下载失败告警
- 写回多维表格失败告警

容量阈值：

| 阈值 | 记录数 |
| --- | --- |
| 50% | 10000 / 20000 |
| 75% | 15000 / 20000 |
| 90% | 18000 / 20000 |

建议先写入状态文件，后续再接飞书通知：

```text
runtime/alerts.json
```

### 5.6 多维表格配置项

`.env.example` 和 README 需要新增：

```text
FEISHU_BITABLE_APP_TOKEN=
FEISHU_BITABLE_TABLE_ID=
FEISHU_BITABLE_VIEW_ID=
FEISHU_BITABLE_FIELD_NAME=名字
FEISHU_BITABLE_FIELD_DESCRIPTION=描述
FEISHU_BITABLE_FIELD_FILE=文件
FEISHU_BITABLE_FIELD_SOURCE=来源
FEISHU_BITABLE_FIELD_USABLE=是否可用
FEISHU_BITABLE_FIELD_STATUS=处理状态
FEISHU_BITABLE_MAX_RECORDS=20000
FEISHU_BITABLE_WARN_RATIOS=0.5,0.75,0.9
YQS_AI_TIMEOUT_SECONDS=120
YQS_BITABLE_BATCH_LIMIT=100
```

## 6. 需要修改的内容

### 6.1 修改 runner 流程

当前 `python/deerflow_runner.py` 的直接流程是：

```text
查看状态 -> 压缩 raw -> AI 识图 -> 查看状态 -> 飞书通知
```

后续要改为：

```text
查看状态
  -> 从飞书多维表格拉取待处理记录
  -> 下载附件到 images_raw
  -> 保存 metadata
  -> 压缩 raw
  -> AI 结合 metadata 识图
  -> 沉淀到本地资产库
  -> 写本地清单
  -> 写回多维表格
  -> 告警检查
  -> 返回 DeerFlow 摘要
```

### 6.2 修改 yqs_deerflow_tools

当前工具：

```text
get_repository_status
compress_raw_images
run_glm_recognition
append_md_table_to_xlsx
send_feishu_summary
```

需要新增工具：

```text
sync_bitable_records
download_bitable_attachments
write_back_bitable_results
check_bitable_alerts
```

也可以先封装成一个工具：

```text
sync_bitable_to_raw
```

第一阶段建议只加一个 `sync_bitable_to_raw`，降低改动风险。

### 6.3 修改 AI 识别 metadata

当前 `recognition_worker.build_metadata()` 只传：

```text
source_path
resolution
file_size
```

后续要追加多维表格字段：

```text
record_id
名字
描述
来源
是否可用
原始附件名
```

AI 识别时要结合这些字段，而不是只看图片。

### 6.4 修改 image_recognition prompt

当前 prompt 中要弱化或删除这些审核式表达：

```text
是否适合进入正式素材库
如果不适合，说明原因和处理建议
不建议使用
不要自动决定正式入库
```

改为 PPT 调用导向：

```text
你的任务是为 PPT 图片资产库生成可调用描述。
默认输出正向用途建议，使用“适合用于…… / 可用于…… / 推荐用于……”。
除非图片严重损坏、明显无关或无法识别，不要输出“不适合用于……”。
是否可用来自飞书多维表格字段，AI 不负责审核或改变该字段。
```

建议输出字段调整为：

```text
suggested_filename
asset_name
image_content
ppt_usage
caption
category
tags
source
usable
usage_note
status
```

其中 `category` 固定为：

```text
背景 / 内容 / 概念 / 信息
```

### 6.5 修改 material_writers

当前 `python/material_writers.py` 写入本地：

```text
案例素材清单_表格.md
案例素材清单.xlsx
素材分辨率.csv
```

后续需要：

- 支持写入 `图片分类`
- 支持写入 `PPT适用场景`
- 支持写入 `关键词`
- 支持保留多维表格的 `名字 / 描述 / 来源 / 是否可用`
- 支持将 `record_id` 传给写回模块

注意：

本地 Excel 的字段可以和多维表格字段不完全一致，但必须能追踪回 `record_id`。

### 6.6 修改 README

README 当前写的是：

```text
同事把图片发给飞书 Bot，图片会自动进入 images_raw。
```

后续要改成：

```text
同事在飞书多维表格上传图片并填写名字、描述、来源、是否可用。
系统从多维表格拉取待处理记录，下载附件到本地并执行处理。
```

也要说明：

- 飞书 Bot 收图可以保留为兼容入口，但不是主入口。
- 主入口是飞书多维表格。

## 7. 是否可用字段的正确处理规则

`是否可用` 的权威来源是飞书多维表格。

建议规则：

```text
是否可用 = y
  下载图片
  压缩
  AI 识别
  写入本地资产库
  写回处理结果

是否可用 = n
  可以同步 metadata
  可以记录为“已跳过”
  不进入 PPT 可调用资产库
  不需要 AI 审核原因
```

注意：

- AI 不判断 `是否可用`。
- Dashboard 不新增人工审核动作。
- 项目只按表格字段执行。

## 8. 告警规则

### 8.1 多维表格容量告警

飞书多维表格上限按 20000 记录设计。

告警点：

```text
50% = 10000
75% = 15000
90% = 18000
```

告警内容建议：

```text
当前多维表格记录数 15000，已达到 75% 容量，请准备归档或拆分子表。
```

### 8.2 AI 识别超时告警

建议默认：

```text
单张图片 AI 识别超过 120 秒，记录超时告警。
```

### 8.3 拉取量过大告警

建议默认：

```text
单次待处理记录超过 100 条，发送提醒。
单次附件总量过大时，发送提醒。
```

第一阶段可以只做记录数告警，附件总量后续再做。

## 9. 推荐实施顺序

### 阶段 1：只做多维表格拉取到本地

目标：

```text
从多维表格读取记录，下载附件到 images_raw，并保存 metadata。
```

要改：

- 新增 `python/feishu_bitable_client.py`
- 新增 `python/bitable_sync.py`
- 新增配置项
- 新增 `runtime/bitable_file_map.json`
- 在 Dashboard / DeerFlow 状态里展示拉取数量

暂时不改 AI prompt。

### 阶段 2：让 AI 使用多维表格字段

目标：

```text
AI 识图时结合名字、描述、来源、是否可用。
```

要改：

- 修改 `recognition_worker.build_metadata()`
- 修改 `kimi_client.py` / `glm_client.py` 的 prompt 输入
- 修改 `rules/image_recognition.md`
- 修改测试，确认 metadata 能进入识图请求

### 阶段 3：写回多维表格

目标：

```text
处理结束后将状态、路径、AI描述、PPT适用场景、分类、错误写回飞书多维表格。
```

要改：

- `feishu_bitable_client.update_record()`
- `bitable_sync.write_back_result()`
- runner 在每条记录成功/失败后写回

### 阶段 4：本地资产库沉淀

目标：

```text
将可用图片沉淀到 case_materials/assets，并按四类分类。
```

要改：

- 新增资产归档函数
- 修改 `material_writers.py`
- 新增本地 metadata 索引

### 阶段 5：告警和子表拆分

目标：

```text
容量、超时、拉取量告警；数据量大后支持子表拆分。
```

要改：

- 新增 `python/alerts.py`
- 新增容量检查
- 新增超时记录
- 后续支持按分类或时间拆分多维表格

## 11. 后续你需要准备的内容

在真正开始开发多维表格拉取前，你需要准备：

```text
1. 飞书多维表格 app_token
2. table_id
3. view_id（如果要只拉某个视图）
4. 字段名称是否完全一致：
   名字 / 描述 / 文件 / 来源 / 是否可用
5. 处理状态字段是否要现在添加
6. 是否可用字段的取值是否固定为 y / n
7. 哪些记录算待处理：
   处理状态为空
   或处理状态 = 待处理
   或所有新记录
8. 处理完成后是否删除多维表格附件，还是只写回本地路径
```

建议你优先确定第 7 点和第 8 点。

## 12. 结论

当前项目已经有本地处理流水线，后续不需要推翻。

真正要新增的是：

```text
飞书多维表格读取
附件下载
表格字段 metadata 保存
处理结果写回
容量/超时/拉取告警
```

真正要修改的是：

```text
runner 启动顺序
AI prompt 口吻
recognition metadata
素材清单字段
README 使用说明
```

最重要的产品边界：

> 本项目不做图片审核系统。多维表格提供素材信息和是否可用字段；本项目负责把这些图片拉到本地，压缩、识别、分类、沉淀，并生成适合 PPT 调用的资产描述。
