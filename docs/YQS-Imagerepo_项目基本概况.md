# YQS-Imagerepo 项目基本概况

更新时间：2026-07-02

## 一句话说明

YQS-Imagerepo 是一套面向公司 PPT、售前方案和案例材料的图片素材沉淀系统。它以飞书多维表格/表单作为素材入口，把同事上传的图片和人工说明拉到本地，经过压缩、AI 识图、语义化命名、分类归档后，沉淀成可被业务人员和后续自动化流程复用的 PPT 图片资产。

## 当前定位

这个项目不是图片审核系统，也不是单纯的文件夹整理工具。

它的核心定位是：

- 飞书多维表格负责收集项目、图片名称、描述、来源、来源二级分类和商用确认。
- 本地 Python 服务负责拉取附件、压缩图片、调用 AI 识图、写入素材清单和移动成品图片。
- Dashboard 负责给管理员查看状态、手动压缩、启动识图、检查相似图片和一键托管处理。
- DeerFlow 负责通过对话入口触发固定素材处理流程。
- `case_materials/` 是最终正式素材库，用于后续 PPT 调用和素材复用，并可通过“回显”同步到飞书知识库。

## 核心处理流程

标准流程如下：

```text
同事填写飞书表单并上传图片
  -> 项目读取飞书多维表格记录
  -> 只下载“是否可商用 = 是”的图片附件
  -> 保存到 image_compressor/images_raw/
  -> 压缩到 image_compressor/images_compressed/
  -> AI 结合图片和表格字段识图
  -> 生成中文语义文件名、PPT 使用场景、图注和关键数据
  -> 写入 案例素材清单_表格.md / 案例素材清单.xlsx / 素材分辨率.csv
  -> 移动成品图到 case_materials/
```

其中 `项目` 字段支持两种归档方式：

| 项目字段 | 归档结果 |
| --- | --- |
| 有项目名 | 进入 `case_materials/项目/` |
| 留空 | 进入 `case_materials/通用素材库/` |

## 主要模块

| 模块 | 位置 | 职责 |
| --- | --- | --- |
| 飞书多维表格客户端 | `python/feishu_bitable_client.py` | 获取字段、视图、记录，下载附件；结构写入需显式 `--apply` |
| 多维表格同步 | `python/bitable_sync.py` | 读取表格记录，过滤不可商用记录，下载图片，保存 metadata |
| 图片压缩 | `python/compress.py` | 把 raw 图片压缩为适合 AI 识别和 PPT 使用的版本 |
| AI 识图 | `python/recognition_worker.py` | 调用 Kimi/GLM 识图，结合表格字段生成素材结构化信息 |
| 清单写入 | `python/material_writers.py` | 更新 Markdown 清单、Excel 清单和分辨率 CSV |
| 飞书知识库回显 | `python/feishu_knowledge_base.py` | 把 `case_materials/` 按本地目录结构同步到飞书知识库 |
| Dashboard 服务 | `python/server.py` + `dashboard/` | 提供本地可视化页面和操作 API |
| DeerFlow 编排 | `python/deerflow_runner.py`、`python/yqs_deerflow_tools.py`、`deer-flow/` | 通过对话或一键托管执行固定处理流程 |
| 识图规则 | `rules/image_recognition.md` | 约束 AI 以 PPT 图片资产调用为目标输出 |

## 已实现能力

当前项目已经具备：

- 飞书多维表格作为主输入源
- 表单上传图片和人工字段
- 表格字段读取和附件下载
- `是否可商用` 作为入库开关
- `项目` 字段作为正式归档依据
- `来源二级分类` 字段进入 AI metadata 和来源/版权输出
- 图片压缩和 raw/compressed 队列管理

- AI 结合表格字段生成 PPT 用途描述

- Markdown、Excel、CSV 三类素材清单
- 按项目归档到 `case_materials/`
- Dashboard 状态查看、压缩、识图、相似图检查和一键托管
- DeerFlow 对话中通过“启动”触发完整处理流程
- 飞书群聊发送 `表单` 返回多维表格表单链接
- 飞书群聊发送 `回显` 把 `case_materials/` 同步到飞书知识库，目录结构尽量和本地一致
- 飞书群聊普通消息默认静默，@ 机器人或发送明确命令才触发回复
- token 用量记录和容量告警文件

## 当前注意事项

1. 当前飞书表单使用 `是否可商用 = 是/否`，旧文档里部分地方仍写 `有/无`。代码两种都兼容，但对同事说明应统一成 `是/否`。
2. 当前 `来源` 选项包含 `AI生成` 和 `渲染`，旧文档里部分地方只写了 `AI`，需要统一表述。
3. `runtime/bitable_sync_state.json` 里仍保留一次旧状态文案，显示“跳过同步”，但当前配置检查结果是多维表格已配置。这是历史状态文件，不代表现在无法读取表格。

## 建议下一步

短期建议先做三件事：

1. 运行 `python3 python/setup_bitable.py` 做只读校验；不要让脚本修改当前手动维护的提交表单。
2. 把旧版部署文档中残留的 `项目-名字`、`有/无`、`feishu_trans_repo` 等旧描述统一到当前多维表格流程。
3. 在确认要处理当前记录后，再执行一次完整同步和处理流程，让附件进入素材库闭环。
